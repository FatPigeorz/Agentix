A **deployment** is a host-side backend that provisions sandboxes. The
user picks one with `agentix deploy <name> --image …`; in code, you
instantiate the class and pass it to `session(...)`. The framework
ships three built-ins — `local` (Docker), `daytona`, `e2b` — and any
`pip install agentix-deployment-*` package adds another.

## Using one

<Tabs>
  <Tab title="CLI">
    ```bash
    # Each named backend is just an entry-point under agentix.deployment.
    agentix deploy local --image my-agent:0.1.0
    agentix deploy fly   --image my-agent:0.1.0    # after `pip install agentix-deployment-fly`
    ```
  </Tab>
  <Tab title="Code">
    ```python
    from agentix import RuntimeClient, SandboxConfig
    from agentix.deployment.base import session
    from agentix.deployment.docker import DockerDeployment

    deployment = DockerDeployment()
    config = SandboxConfig(image="my-agent:0.1.0")
    async with session(deployment, config) as sandbox:
        async with RuntimeClient(sandbox.runtime_url) as c:
            ...
    ```
  </Tab>
</Tabs>

`session(deployment, config)` is a free function — composition over
inheritance. It calls `create()` on entry and `delete()` on exit.

## Writing one

A deployment backend is a class that satisfies the three-method
`Deployment` Protocol. **Structural type — no inheritance, no base
class.**

<Tabs>
  <Tab title="Plugin source">
    ```python my_deploy/__init__.py
    import os
    from agentix.deployment.base import Sandbox
    from agentix.idents import SandboxId
    from agentix.models import SandboxConfig, SandboxInfo

    class FlyDeployment:
        def __init__(self) -> None:
            # Backends take NO constructor arguments — read config from env.
            self._token = os.environ["FLY_API_TOKEN"]

        async def create(self, config: SandboxConfig) -> Sandbox: ...
        async def delete(self, sandbox_id: SandboxId) -> None: ...
        async def get(self, sandbox_id: SandboxId) -> SandboxInfo: ...
    ```
  </Tab>
  <Tab title="pyproject.toml">
    ```toml
    [project]
    name = "agentix-deployment-fly"
    version = "0.1.0"

    [project.entry-points."agentix.deployment"]
    fly = "my_deploy:FlyDeployment"
    ```
  </Tab>
</Tabs>

After `pip install`, `agentix deploy fly --image my-agent:0.1.0` is
live with no framework changes.

`isinstance(FlyDeployment(), Deployment)` returns True on its own —
the framework's runtime check is structural, so a backend can satisfy
the interface without coupling to the framework class hierarchy.

## How discovery works

The CLI walks `importlib.metadata.entry_points(group="agentix.deployment")`
when `agentix deploy <name>` runs and instantiates the matching class
with **no constructor arguments**. Backends read API keys, regions,
and other configuration from environment variables in `__init__`.

Two distributions registering the same name raise `PluginConflictError`
with both dist labels — the CLI accepts one `<name>` argument, so
silent last-wins would hide a stale install.

## Configuration

Backend config is environment-variable based, by convention:

| Variable                          | Used by              | Purpose                                  |
| --------------------------------- | -------------------- | ---------------------------------------- |
| `DAYTONA_API_KEY`                 | `daytona`            | API auth                                 |
| `E2B_API_KEY` / `E2B_TEMPLATE_ID` | `e2b`                | API auth + template                      |
| `AGENTIX_BIND_PORT`               | runtime server       | Bind port (default 8000)                 |

Read your backend's env vars in `__init__` and fail fast if any are
missing — surfacing the error at CLI time is friendlier than waiting
for the first `create()` to fail.

## When you write one

A new sandbox host — Fly, Modal, Kubernetes, a managed-sandbox vendor,
a local-process backend for tests. Backends that talk to different
APIs but share a Protocol are exactly the shape this axis was designed
for.
