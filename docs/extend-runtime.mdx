title: Extend the runtime
description: Host-side hooks for tracing, wire patterns, spec resolvers, and the CLI — what's pluggable, what isn't, and how to extend each.
---

The two **plugin axes** ([Namespace](/namespace) and
[Deployment](/deployment)) exist because they cross the host↔sandbox
boundary. Everything else lives on the host as plain Python — you
extend it by importing and calling, not by `pip install`. This guide
walks each host-side hook.

## Trace pub/sub — fan-out observability

Every namespace's `agentix.trace.emit(kind, payload)` call fans out to
every subscribed handler. Handlers attach once at runtime startup via
an **installer** — a callable the framework invokes during lifespan:

```python title="my_otel_handler/__init__.py"
from opentelemetry import trace as otel
from agentix.trace import subscribe

def install() -> None:
    tracer = otel.get_tracer("agentix")

    def handler(kind, payload, call_id, source):
        with tracer.start_as_current_span(f"agentix.{kind}") as span:
            span.set_attribute("agentix.call_id", call_id or "")
            span.set_attribute("agentix.source", source or "")
            for k, v in payload.items():
                span.set_attribute(f"agentix.payload.{k}", v)

    subscribe(handler)
```

Call `install()` from your own startup code:

```python
import my_otel_handler
my_otel_handler.install()
```

Handler errors are caught and logged — tracing never breaks a rollout.
But handlers that **block** will slow tracing for the entire runtime;
offload heavy work (network I/O, file flushes) to a background thread
or async task inside the handler body.

<Tip>
  **Why isn't this an entry-point axis?** Trace handlers run host-side,
  not in the sandbox. Plugin discovery via `pip install` is for things
  that cross the process boundary. A host-side hook is one function
  call — `import` and `subscribe(my_handler)` is enough. No framework
  dispatch needed.
</Tip>

## Call shapes — three built-ins, detected by signature

Three call shapes are detected from a method's signature by
`agentix.dispatch.detect_shape(sig)`:

| Shape    | Signature                                                          |
| -------- | ------------------------------------------------------------------ |
| `unary`  | plain return (`-> T`)                                              |
| `stream` | `-> AsyncIterator[T]`                                              |
| `bidi`   | one `AsyncIterator[U]` param + `-> AsyncIterator[V]` return        |

Detection runs at `Dispatcher.bind` time on the server and on every
`c.remote(...)` on the client. If a fourth shape ever becomes
necessary, edit `detect_shape` plus the matching branches in
`Dispatcher.bind` and `RuntimeClient.remote` — no plugin hook.

## Spec resolvers — internal ordered list

`agentix build <spec>` accepts four spec shapes today: a local path
(`./primitives/bash`), a docker image ref
(`docker.io/me/agent:1.0`), a short name resolved against the repo
(`bash`), and a PyPI dist (`agentix-bash`). The resolution chain lives
in `agentix/cli/_resolve.py`:

```python
RESOLVERS = [
    PathResolver(priority=100),
    ImageResolver(priority=90),
    LocalRepoResolver(priority=50),
    PypiResolver(priority=10),
]
```

The CLI walks the list in priority order; the first resolver whose
`resolve()` returns non-`None` wins. **Add a new spec shape by editing
that file** — extending the CLI's input grammar is a framework concern,
not a per-wheel one.

## A new `agentix <verb>` CLI

The four built-in subcommands (`build`, `install`, `deploy`, `check`)
are hardcoded in `agentix/cli/__init__.py`. The dispatcher is *not* a
plugin surface. Third parties that want a new top-level verb should
ship a separate `console_scripts` binary in their wheel:

```toml title="pyproject.toml"
[project.scripts]
agentix-yourcmd = "your_package:main"
```

Users invoke it directly: `agentix-yourcmd …`. The convention is
`agentix-<verb>` so it sorts alphabetically next to the framework
verbs in shell completion.

<Tip>
  **Why not a plugin?** The CLI dispatcher is a thin argparse handler
  with no per-subcommand state. Loading every wheel's subcommands at
  every CLI invocation would slow down `agentix --help` and turn a
  static argparse tree into a dynamic registry. A `console_scripts`
  binary is the same UX with none of that cost.
</Tip>

## Summary

| Surface                        | Mechanism                          | Where to add it                              |
| ------------------------------ | ---------------------------------- | -------------------------------------------- |
| Trace pub/sub                  | `trace.subscribe(fn)` at startup   | Your own host-side startup code              |
| Call shapes                    | `agentix.dispatch.detect_shape`    | `agentix/dispatch.py` (framework PR)         |
| Spec resolvers                 | In-tree ordered list               | `agentix/cli/_resolve.py` (framework PR)     |
| A new `agentix <verb>`         | Separate `console_scripts` binary  | Your own wheel — `agentix-<verb>` convention |

If you find yourself wanting to ship a new wire pattern or spec
resolver as a plugin, file an issue first — that's a framework-level
design decision, not a wheel-level one.
