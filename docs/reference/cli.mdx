The four built-in subcommands are hardcoded in
`agentix/cli/__init__.py`. Third parties that want their own top-level
verb should ship a `console_scripts` binary (`agentix-yourcmd`) rather
than registering against the central dispatcher.

```bash
agentix --help
```

## `agentix build`

Build one namespace image, or bundle several into a single image. The
verb is the same; passing one spec produces a single-namespace image,
passing many produces a bundle.

```bash
# One namespace — tag auto-derived from pyproject.toml
agentix build primitives/bash
agentix build bash --dry-run
agentix build bash -o my-bash:dev

# Many namespaces — explicit -o required
agentix build bash files claude-code -o my-agent:0.1.0
agentix build primitives/bash ./my-experimental-namespace -o demo:dev
agentix build bash files --dry-run -o sandbox:dev
```

Each spec accepts one of:

- **Path:** `./primitives/bash` or any directory with a `pyproject.toml`.
- **Short name:** looked up under the repo's `primitives/`.
- **PyPI dist:** `agentix-bash` (currently surfaces NotImplementedError;
  fetch path is stubbed).

`-o / --output / --tag` is the output image tag. With one spec it's
optional (auto-derived as `agentix/<short>:<version>`). With multiple
specs it's required. `--dry-run` stages the docker build context to
`./build/<name>/` and prints the path instead of invoking docker.

The runtime discovers every bundled namespace via
`importlib.metadata.entry_points(group="agentix.namespace")` at startup —
no bundle disposition file is needed.

## `agentix deploy`

Provision a sandbox.

```bash
agentix deploy <backend> --image <tag>
agentix deploy local --image my-agent:0.1.0
agentix deploy local --image my-agent:0.1.0 --detach
agentix deploy daytona --image docker.io/me/my-agent:0.1.0    # currently NotImplementedError
```

`<backend>` is the name of any registered `agentix.deployment` plugin.
The three built-in backends are `local`, `daytona`, and `e2b`; any
`pip install agentix-deployment-*` package adds its own.

By default the command stays in the foreground: prints the sandbox's
`runtime_url`, waits for Ctrl-C, then tears the sandbox down.
`--detach` exits after `create()` and just prints the handle so the
user can manage the sandbox lifetime themselves.

- `--base` — base task image (default `ubuntu:24.04`)
- `--runtime` — runtime image ref (default `agentix/runtime:latest`)
- `--detach` — exit immediately after sandbox creation

## `agentix check`

List installed namespaces and smoke-import each one.

```bash
agentix check
```

With the entry-point model, namespace stub-impl drift is impossible
(one class, the methods are the implementation). `agentix check` is a
fast health check: every namespace declared under `agentix.namespace`
actually imports without error. Non-zero exit code if any failed.

## Configuration via environment

Most plugin-specific config reads from environment variables, not
command-line flags.

| Variable                          | Used by                    | Purpose                                |
| --------------------------------- | -------------------------- | -------------------------------------- |
| `AGENTIX_BIND_PORT`               | `start` / runtime server   | Bind port (default 8000)               |
| `AGENTIX_UPLOAD_ROOT`             | `agentix.files` namespace  | Sandbox-side root for file I/O         |
| `DAYTONA_API_KEY`                 | `daytona` deployment       | API auth                               |
| `E2B_API_KEY` / `E2B_TEMPLATE_ID` | `e2b` deployment           | API auth + template                    |
