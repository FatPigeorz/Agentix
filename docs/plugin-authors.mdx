---
title: Writing Agentix plugins
description: Extend Agentix along either of two entry-point-discovered axes — namespaces and deployments.
---

Agentix has two plugin axes — the things that cross the host↔sandbox
boundary. Each is a Python entry-point group; every plugin is a normal
pip-installable distribution. To contribute a plugin you write two
things:

1. A class implementing the axis's `Protocol`.
2. One `[project.entry-points."agentix.<axis>"]` block in your
   `pyproject.toml`.

`pip install your-plugin` makes the plugin live to every Agentix
installation in the same environment. No framework patch, no config
file, no decorator at import time.

The framework's own builtins ship via the same pattern, so every example
below has an in-tree precedent. Host-side extension hooks (trace sinks,
the three built-in wire patterns, internal spec resolvers, the CLI
dispatcher) are plain Python APIs — they do not participate in
entry-point discovery and are documented in
[The two plugin axes](/axes#host-side-extension-hooks).

## Axis index

| Axis                         | Group                | Semantics                         | Built-ins                                  |
| ---------------------------- | -------------------- | --------------------------------- | ------------------------------------------ |
| [Namespaces](#namespaces)    | `agentix.namespace`  | discover, lazy-load on first call | (third-party only)                         |
| [Deployments](#deployments)  | `agentix.deployment` | select-one by name                | `local` / `daytona` / `e2b`                |

## Namespaces

A namespace is a class whose `@staticmethod` methods are the callable
surface. Methods carry the real implementation; the class is a
namespace.

```python src/agentix/myagent/__init__.py
from agentix.namespace import Namespace

class MyAgent(Namespace):
    @staticmethod
    async def run(instruction: str) -> str:
        ...
```

```toml pyproject.toml
[project]
name = "agentix-myagent"
version = "0.1.0"

[project.entry-points."agentix.namespace"]
myagent = "agentix.myagent:MyAgent"

[tool.hatch.build.targets.wheel]
packages = ["src/agentix"]
```

After `pip install`, callers do `from agentix.myagent import MyAgent`
and use `await c.remote(MyAgent.run, instruction=...)`. See
`primitives/bash` and `primitives/files` for working examples.

## Deployments

A deployment manages sandbox lifecycle. Structural type — implement
the three methods, no inheritance.

```python my_deploy/__init__.py
from agentix.deployment.base import Sandbox
from agentix.idents import SandboxId
from agentix.models import SandboxConfig, SandboxInfo

class FlyDeployment:
    def __init__(self) -> None:
        # Backends take NO constructor arguments. Read config from env.
        import os
        self._token = os.environ.get("FLY_API_TOKEN")

    async def create(self, config: SandboxConfig) -> Sandbox: ...
    async def delete(self, sandbox_id: SandboxId) -> None: ...
    async def get(self, sandbox_id: SandboxId) -> SandboxInfo: ...
```

```toml pyproject.toml
[project.entry-points."agentix.deployment"]
fly = "my_deploy:FlyDeployment"
```

After install, `agentix deploy fly --image my-agent:0.1.0` works.
Conflicts (two dists registering the same name) raise
`PluginConflictError` with both dist labels.

## Testing your plugin

Production discovery happens via `importlib.metadata`. For unit tests,
each axis exposes an in-process `register_*` helper that the framework's
own tests use to avoid actually installing distributions:

| Axis        | Imperative helper                                              |
| ----------- | -------------------------------------------------------------- |
| Namespaces  | `server.registry.register(...)` (in-process Dispatcher binding) |
| Deployments | `agentix.deployment.base.register_deployment(name, cls)`       |

`agentix._plugin.Registry[T].reset()` clears per-test state.

After installing your plugin into a real environment, `agentix check`
smoke-imports every installed namespace; mismatched entries surface
there. There is no `agentix plugins` command — the framework only ships
two plugin axes and each has its own listing path (`agentix check` for
namespaces; `agentix deploy --help` lists deployments).
